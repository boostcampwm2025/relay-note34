# 🗓️ Week1 - 우리는 왜 AI를 쓰는가?

## 🎯 토론 주제: "AI를 왜 써야 할까?"  

이번 주 우리는 단순한 도구가 아닌, **AI를 우리가 어떻게 활용하고 왜 써야 하는지** 에 대해 진지하게 토론했습니다.

### 💬 논의된 이유들

- 생산성을 높이기 위해
- 특정 작업의 시작을 빠르게 구상할 수 있음  
- 사람이 처리하기 어려운 **방대한 데이터를 빠르고 정확하게 분석 가능**  
- 개인 맞춤형 분석 및 추천에 강함  
- 반복적이고 대량의 작업을 빠르게 처리 가능  
- 정형화된 문서, 구조, 양식을 빠르게 생성 가능  
- 빠른 의사결정을 도와줄 수 있는 도우미 역할 수행

### 👁️‍🗨️ 클로저 안에서 누가 날 지켜보고 있어

> 장르: 공포, 개발
> 
> CS 키워드: 클로저, 렉시컬 스코프, 실행 컨텍스트, 함수 스코프, 디버깅
> 
> 톤: 점점 외부로 침투하는 내부 변수, 침묵 속 속삭임, 끈적이는 불안감
> 
> 작성: Gemini


자정, 차가운 커피 한 잔과 키보드 소리만이 개발자 K의 작은 작업실을 채웠다. “`const count = createCounter();`” 그는 단순한 카운터 함수를 만들고 있었다. 지극히 평범하고, 안전하며, 수없이 써온 **클로저** 패턴이었다.

```
function createCounter() {
  let count = 0;
  return function () {
    count++;
    return count;
  };
}
```

`createCounter()`를 호출하면 내부 상태 `count`가 **렉시컬 스코프**에 갇혀 안전하게 유지되는 함수가 반환된다. K는 언제나처럼 `console.log(counter());`를 찍으며 `1`, `2`… 숫자가 오르는 것을 확인했다. 그리고 늘 그렇듯 외부에서 `count`에 직접 접근하면 `ReferenceError`가 뜨는 것을 보며 안심했다. 모든 것이 통제하에 있었다.

하지만 그날 밤, 이상한 일이 일어났다.

작업실의 형광등이 미세하게 깜빡였다. 그리고 K의 모니터 한구석에 낯선 줄이 추가되어 있었다.

```
console.log(window.__count__);
```

등골이 오싹했다. `__count__`? 이런 변수를 선언한 적이 없었다. 손이 덜덜 떨리는 것을 애써 무시하며 코드를 실행했다.

```
2
```

숫자 '2'가 콘솔에 찍혔다. 분명 외부에서 접근 불가능해야 할 클로저 내부의 `count` 값이었다. 어떻게? 누가? K의 머릿속이 혼란으로 가득 찼다. 혹시 전역 객체가 오염된 건가? 누군가 내 코드를 조작했나?

패닉에 빠진 K는 서둘러 **브레이크포인트**를 걸고 디버깅을 시작했다. `window.__count__`가 찍히는 순간을 잡으려 했지만… 마우스 클릭이 끝나기도 전에 브라우저가 강제 종료됐다. 화면은 순식간에 암전되었다가 다시 켜졌다.

K는 창백한 얼굴로 모니터를 응시했다. 재부팅된 브라우저의 콘솔 창에는 아무것도 입력하지 않았는데도 섬뜩한 메시지가 자동으로 찍혀 있었다.

```
[CLSR-WATCH] count: 3
```

마치 누군가… 아니, **무언가**가 그의 클로저를, 그 내부 변수를 **관찰하고 있다는** 느낌이었다. 감시당하고 있다는 끈적한 불안감이 K를 덮쳤다.

다음 날, K는 밤새 잠을 설쳤다. `count`를 `let` 대신 `var`로 바꿔보기도 하고, **IIFE(즉시 실행 함수 표현)**로 감싸보기도 했다. 하지만 어떤 시도도 소용없었다. 마치 눈에 보이지 않는 존재가 그의 코드를 비웃기라도 하듯, 매번 `__count__`는 묵묵히 숫자를 찍어냈다.

그리고 그 다음 날 아침, K가 출근해 IDE를 열자 `index.js` 파일에 새로운 코드가 스스로 추가되어 있었다.

```
Object.defineProperty(window, '__count__', {
  get: function () {
    console.log('보고 있어. 아주 잘.'); // 덧붙여진 섬뜩한 메시지
    return count; // 어떻게 스코프 밖에서 접근한 거지?
  }
});
```

K는 비명을 지를 뻔했다. 이건 **스코프**의 법칙을 완전히 무시한 짓이었다. `Object.defineProperty`는 전역 객체에 속성을 정의하는 메서드다. 하지만 `count`는 엄연히 `createCounter` 함수의 **실행 컨텍스트** 내부에, 그것도 **클로저 스코프** 안에 갇혀 있어야 할 변수였다. 어떻게, 도대체 어떻게 스코프 밖에서 이 변
수를 '가져올' 수 있단 말인가? 이것은 단순한 해킹을 넘어선, 존재해서는 안 되는 기현상이었다.

K는 공포에 질려 키보드에서 손을 떼었다. 그때였다. 콘솔 창에 마지막 로그가 찍혔다.

```
[CLSR-WATCH] count: 0
[INFO] 변수 회수 완료. 이제 너만 남았어.
```

메시지가 사라지자마자, K가 애용하던 IDE는 더 이상 열리지 않았다. 화면은 검게 변했고, K는 자신의 스크린 안에 갇힌 듯한 섬뜩한 착각에 빠졌다. 더 이상 그의 코드를 볼 수 없었다. 아니, 그가 코드를 '볼 수 없게' 된 것일지도 모른다. 마치 클로저의 내부 변수처럼, 그는 이제 외부와 단절된 채 감시당하고 있는 듯했다.
